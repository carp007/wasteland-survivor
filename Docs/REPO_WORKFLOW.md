# Zip Workflow (How we iterate going forward)

This document is the canonical process for iterating on Wasteland Survivor using **Project source zips** (not remote repo pulls).

## Goals
- Keep iterations fast and low-friction.
- Avoid “which build is the source of truth?” confusion.
- Keep downloads small by excluding heavy/generated folders.

## Source of truth
- **Baseline:** the latest Project file uploaded by the user: `wasteland-survivor.zip`

Notes:
- Threads have been unreliable at pulling from remote repo (including “main.zip” fallback).
- Treat the Project zip as canonical for iteration baselines until/unless the user switches back.

## Standard iteration loop

### 1) Baseline
- The user uploads the latest full folder as `wasteland-survivor.zip` into the Project files.
- The AI starts from that zip (extract locally and modify).

### 2) Deliverable from the AI
The default deliverable is a **single zip** of the project with all updated code/docs, but **excluding**:
- `Assets/`
- `.godot/`
- `.git/`

Notes:
- `Assets/` is excluded to keep downloads small (assume the user keeps Assets locally).
- `.godot/` is generated by Godot and should not be shipped in zips.
- `.git/` should never be shipped in zips.

If a change requires new/updated files under `Assets/`, the AI must:
- explicitly list the required files/paths in the response, and
- keep them out of the main project zip (ideally provide a separate small asset bundle only when truly needed).

### 3) User local apply steps
You will:
1) **Delete local project contents** **except** these folders (keep them):
   - `Assets/`
   - `.godot/`
   - `.git/`
2) **Drop the contents of the AI zip** into the project folder.
3) **Run/build and validate** the update.

### 4) Baseline update policy
- In most cases, even if there are bugs, you will **upload the updated full folder** again as `wasteland-survivor.zip`.
- Only in cases where something has gone massively wrong would you not do this.
- The expectation is we iterate from the latest zip each time and fix forward.

## Release notes / versioning expectations
Whenever a zip is delivered, the AI should also update:
- `VERSION.txt` (bump build id)
- `CHANGELOG.md` (short entry)
- any relevant docs in `Docs/` (especially if workflow changes)

## Packaging checklist (AI)
Before creating the downloadable zip, the AI should ensure:
- Project compiles (or changes are small and compile-safe).
- No files from these folders are included in the zip:
  - `Assets/`
  - `.godot/`
  - `.git/`
- Changelog/version/docs updated as needed.

## Assistant process notes (to avoid repeat mistakes)
The assistant should keep a short, living “what works / what doesn’t” list here:
- `Docs/ASSISTANT_PLAYBOOK.md`

If the assistant hits friction with zip extraction, packaging exclusions, or re-learning the workflow, it should update that playbook at the end of the iteration so the next iteration doesn’t repeat the same dead-ends.
