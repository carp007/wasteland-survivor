# Repo Workflow (How we iterate going forward)

This document is the canonical process for iterating on Wasteland Survivor now that the project is in a public GitHub repo.

## Goals
- Keep iterations fast and low-friction.
- Avoid zip "source of truth" confusion.
- Keep downloads small by excluding heavy/generated folders.

## Source of truth
- **Repository:** `https://github.com/carp007/wasteland-survivor.git`
- **Branch:** `main`

### Do we need commit SHAs/tags every time?
No.
- Default behavior is: **work from the latest `main`**.
- If we’re debugging something tricky, bisecting, or you want reproducibility, you can optionally provide a **commit SHA** (or tag). When provided, that specific revision becomes the baseline.

## Standard iteration loop
### 1) Baseline
- The AI starts from the **latest `main`** (unless you specify a SHA/tag).

### 2) Deliverable from the AI
The default deliverable is a **single zip** of the project with all updated code/docs, but **excluding**:
- `Assets/`
- `.godot/`
- `.git/`

Notes:
- `Assets/` is excluded to keep downloads small.
- `.godot/` is generated by Godot and should not be shipped in zips.
- `.git/` should never be shipped in zips.

If a change requires new/updated files under `Assets/`, the AI must:
- explicitly list the required files/paths in the response, and
- keep them out of the main project zip (ideally provide a separate small asset bundle only when truly needed).

#### If zip delivery ever becomes inconvenient
If we ever want to reduce friction further, we can switch to a patch-based flow:
- AI delivers a `git diff` patch against `main`.
- You apply it locally and push.
- If you still want a zip snapshot, you generate it locally from git (`git archive`) so it’s always consistent.

### 3) Your local apply steps
You will:
1) **Delete local project contents** **except** these folders (keep them):
   - `Assets/`
   - `.godot/`
   - `.git/`
2) **Drop the contents of the AI zip** into the project folder.
3) **Run/build and validate** the update.

### 4) Repo update policy
- In most cases, even if there are bugs, you will **push the updated code to `main`**.
- Only in cases where something has gone massively wrong would you not push.
- The expectation is we’ll pull `main` next iteration and fix forward.

## Release notes / versioning expectations
Whenever a zip is delivered, the AI should also update:
- `VERSION.txt` (bump build id)
- `CHANGELOG.md` (short entry)
- any relevant docs in `Docs/` (especially if workflow changes)

## Packaging checklist (AI)
Before creating the downloadable zip, the AI should ensure:
- Project compiles (or changes are small and compile-safe).
- No files from these folders are included in the zip:
  - `Assets/`
  - `.godot/`
  - `.git/`
- Changelog/version/docs updated as needed.
